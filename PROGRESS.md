# 量词大冒险 — 进度记录

一年级华文「5.3 认识各类词语——量词」的自主学习 + 闯关游戏。改编自 `../../mario-game/index.html`
（Phaser 3 像素闯关引擎），题库换成 17 个量词，前面加了强制学习 slides + 分数达标闸门。

## 怎么玩（本地试玩）

```
cd teaching-tools/tahun1-bc-liangci
python3 -m http.server 8744
```
浏览器打开 http://localhost:8744/ 。

## 整体流程
0. **开始页**：像素风主题页，点「开始游戏」才进选班级——不是一进网页就是学生名单
1. **选班级 + 选人**：三个选项——1I / 1G（沿用《一年级华文闯关问答》同一份名单），或「不选名字」
   直接开始（方便其他学校借用这个游戏，不用维护班级名单；这个模式不会提交排行榜）
2. **Slides 学习**（19 页：1 页量词概念介绍 + 17 个量词 + 1 页总结，字体/卡片针对一年级放大，
   每个量词卡片轮换 6 种主题色）：
   - 门槛不分「第一次学习」和「答错重学」，都一样：把 17 个量词、**51 个例子短语**（`SLIDE_PHRASES_PER_WORD`
     = 每词 3 个，如「一座大山」）**逐一点击确认读过**，全部点完「开始闯关游戏」才解锁——没有时间限制
   - 每页至少停留 0.8 秒才能翻下一页（防止狂点跳过）
3. **游戏闯关**：Phaser 版 Mario，一轮固定 **10 题**（题库 107 题随机抽 10，不重复；已排除会有两个
   正确答案的名词，如「葡萄」同时能配「颗」和「串」，见下方「题库多解题」），沿用原本双段跳/坑洞/
   终点旗杆机制，另外加了：
   - 地面怪物（青蛙，巡逻+可踩）+ **天上的怪兽**（老鹰，飞行巡逻+轻微上下浮动，可踩可撞）
   - 每个问答站除了原本的问答方块+砖块，额外加了**垫脚的矮箱子**（单向平台，只能从上面踩，
     从旁边走过或下面跳都会直接穿过，不会挡住去路）；最高的问答方块档位铺 2 层楼梯
   - **每一轮关卡布局都会重新生成**：问答方块高度（3.6/4.4/5.0 个 TILE 三种随机）、垫脚箱子位置、
     地面怪物/天上老鹰的位置・巡逻范围・初始方向・飞行高度，都是每段关卡各自独立随机
   - 答题选项 4 个各配一种底色（蓝/粉/绿/紫），上下键切换选中时额外加金色发光框
4. **结算**：**必须把 10 题都答完，而且正确率 ≥ 70%**（`PASS_RATE` 常量可调）才算达标——
   只答对 1 题、其余没答就冲去终点不会算过关
   - 达标：可以「再玩一次」（新一轮 10 题）、「重新看资料」、看排行榜，或换人
   - 不达标：只能「重新学习资料」，强制回到 slides 重新点完全部 51 个例子，没有跳过游戏的路径
5. **排行榜**：历史最高分 + 累计挑战次数，阶段一先用 localStorage 模拟，接了 Google Sheet 后端后
   自动变成全班共享；后端有白名单验证（见下方「排行榜安全」）

## 文件结构
- `index.html` — 全部前端代码（开始页/选人/学习/游戏/结算/排行榜六个画面 + Phaser 场景）
- `Code.gs` — 排行榜后端（Google Apps Script，纯 JSON API，不含任何页面），部署方式见 `部署步骤.md`
- `assets/` — 直接复用 `mario-game/assets` 的 SunnyLand 美术素材（同一份文件，非任天堂角色，无版权顾虑）

## 关键设计决定
- **架构**：网页本体部署 Vercel/GitHub（静态站，跟 teaching-tools 其他工具一致）；共享排行榜用 Google Sheets 存资料，但只透过一个「只回传 JSON、学生看不到」的 Apps Script Web App 读写——用户明确要求不要整个网站放在 script.google.com 上面，只接受 Apps Script 当安静的 API 桥梁
- **学习门槛 = 点击确认，不计时**：一开始设计过「翻完 19 页 + 满 5 分钟」的计时门槛，用户反馈「点完卡片本来就要花不少时间了，不需要再计时」，改成不分场景统一用「例子逐一点击确认」当门槛，答错重学时也是重新点一遍全部例子（`confirmedPhrases` 每次进入 slides 都会清空重来）。原本每词 6 个例子（102 个），用户反馈一年级小朋友点太多，改成每词只需点 3 个（`SLIDE_PHRASES_PER_WORD`），门槛降到 51 个——但 `WORDS.phrases` 完整的 6 个短语还留着，题库生成仍然用全部 6 个，只有 slides 显示/门槛用前 3 个
- **每轮 10 题**：`ROUND_SIZE` 常量控制，从最初的 20 题（用户嫌太长）调整为 10 题；关卡长度、怪物/宝石站点都跟着 `ROUND_SIZE` 自动生成
- **题库来源**：17 个量词、每个 6 个四字短语 + 1 个例句 = 119 题候选池，`buildQuestionTemplates()` 会先排除「同一个名词在不只一个量词底下都出现过」的组合（这类名词裸模板「一（ ）NOUN。」会有两个都对的答案，见「题库多解题」），实际候选池 107 题，每轮 `shuffleArr().slice(0,ROUND_SIZE)` 随机抽（已用 node 脚本模拟验证：选项 4 个不重复、答案索引有效、无重复正确答案冲突）
- **达标标准**：必须 10 题都答完（`total === ROUND_SIZE`）且正确率达标（`rate >= PASS_RATE`）两个条件都满足，不能只答 1 题就冲终点
- **踩坑记录：宝石卡进箱子里**——最初每段关卡中间那颗装饰宝石的 x 坐标跟砖块几乎重叠（只差 2px），学生反馈"看起来像卡在箱子里"；修正为明确放在砖块正上方、留出清楚的垂直间距，改用 `{x,y}` 物件取代原本靠 `i%3` 猜高低的写法，逻辑更直观也更不容易再犯同样的错
- **踩坑记录：垫脚箱子反复卡关，最后改成单向平台才解决**——第一版把矮箱子叠在大箱子正上方（纯装饰）；改成放旁边当台阶后，高度公式跟着主方块等比例升高，主方块抽到最高档时垫脚箱自己都高到跳不上去；改成固定安全高度（1.7 个 TILE，理论单跳极限约 2.55）后，又发现箱子底部矮到跟玩家头顶差不多高，正好卡在地面通道正中间，双跳没接好会被弹开卡在原地过不去。最终方案：**垫脚箱做成单向平台**（`processBlockCollision` 只有玩家从上方落下、快贴近箱顶时才当实心地板，从旁边走过或下面跳都直接穿过），箱子再也不可能挡住前进的路，就算跳失败也只是穿过去，从根本上排除了"卡关"这个失败模式，不用再纠结跳跃高度公式怎么调都还是会有边界情况
- **关卡布局改成每轮重新生成**（`generateLevelLayout()`），不再是模块加载时算一次的固定 `const` 数组——原本每段关卡的方块高度/怪物位置/宝石位置都是完全相同的相对偏移，用户反馈「玩起来太单调，好像一直在重复」；改成 `let` + 可重复调用的生成函数，每次 `startNewRound()` 都重新洗一次牌（已用 node 脚本模拟 200 轮验证：垫脚箱不会掉进坑里、位置不会跑出关卡范围）
- **踩坑记录：地面怪物随机巡逻把玩家卡在坑洞前**——加随机化时地面青蛙的巡逻右边界 `x+range` 算出来可能超过坑洞起点（620 关卡宽度里坑在 `+400` 起），玩家反馈"卡住了过不去"，实际原因是青蛙巡逻到坑边把玩家一直撞退，没办法助跑跳过去；修正为巡逻范围改存明确的 `minX`/`maxX`，并强制 `maxX` 不超过 `segX+370`（离坑洞起点留 30px 安全缓冲），已用 node 脚本模拟 5000 次确认 `maxX` 恒小于坑洞起点
- **踩坑记录：终点旗杆那一段没有地面**——地面绘制的 for 循环在楼梯前面就停了（`x < STAIR_START_X - TILE/2`），楼梯和坑洞后面、旗杆终点那一段完全没有再铺地面——不只是画面空，那段地连碰撞体都没有，等于悬空；加了第二段循环从 `GAP_END` 铺到 `LEVEL_WIDTH`
- **题库多解题**：`buildQuestionTemplates()` 生成裸模板「一（ ）NOUN。」时，有些名词在不只一个量词底下都出现过（例如「葡萄」配「颗」是一颗葡萄、配「串」也对；「孩子」配「个」或「群」也都对），这在学习卡片里是正常的一词多量词用法，但拿来出选择题就会有两个都算对的答案。修复方式是统计每个名词出现在几个量词底下，出现超过一次的名词直接跳过、不拿来出题（学习卡片两边都还留着）——不是手动列出冲突名词清单，是自动检测，以后加新量词/新短语也不会漏掉新的冲突组合
- **排行榜安全**：`Code.gs` 原本 `doPost` 接受公开的 `action:'clear'`，任何人对着 Web App 网址发请求就能清空全部排行榜资料；`submitScore` 也没有验证 `cls`/`name`/`score`，理论上可以伪造任意班级/姓名/超高分数提交。修复：① 拿掉公开的 clear 入口，`clearScores()` 只能在 Apps Script 编辑器里手动执行；② `validateSubmission()` 用跟前端同一份 `CLASSES` 白名单验证班级+姓名，`score` 限制在 0~300（10 题满分 100 + 旗杆最高 80 奖励，留余裕）的整数。**前端加/减学生名单时，`Code.gs` 里的 `CLASSES` 常量要手动同步**，不是自动共享同一份资料
- **排行榜 XSS**：`loadBoard()` 原本把后端读回来的 `name`/`cls` 直接拼进 `innerHTML` 字符串，如果排行榜姓名字段被塞入 HTML/script 会在别人打开排行榜页面时执行（存储型 XSS）。改成用 `document.createElement` + `textContent` 逐一组装 DOM 节点，不再用字符串拼接
- **手机触摸滚动**：`touch-action:none` 原本设在 `html, body` 全局，导致手机在学习页（内容常常比视口高）滑不动、按不到下方按钮。改成只设在 `#screen-game`，其他画面手机可以正常滑动

## 已知/待处理
- Vercel 项目的 GitHub 自动部署连接失败过一次（`vercel git connect` 报错，可能是 GitHub App 权限还没授权到这个新仓库），目前是手动 `vercel deploy --prod` 上线的，之后改代码要记得手动重新部署，或请用户去 Vercel 后台补权限
- 学生完整姓名目前写死在前端源码里（`CLASSES` 常量，`Code.gs` 也有一份同步的白名单），公共网页源码可以直接看到两班学生姓名——如果要更在意未成年人隐私，可以考虑改成学号/代称，或整理成教师端可管理的名单，但会牵动选人页 UI 和排行榜显示，是产品决定不是纯 bug，还没跟用户讨论定案
- 没有做移动端触屏控制（跟原版 mario-game 一样，游戏本体只支持键盘操作；学习 slides 页面手机触摸没问题）
- Phaser 完全依赖 jsDelivr CDN（`<script src="https://cdn.jsdelivr.net/...">`），学校网络如果挡 CDN，游戏会整个启动不了；可以考虑把 Phaser 文件下载到 `assets/` 本地化，还没做
- 1400+ 行 HTML/CSS/题库/游戏逻辑全部在一个文件里，后续改动容易互相影响；可以考虑拆成 `data.js`/`ui.js`/`game.js`/`styles.css`，还没做（拆分本身风险较高，要先有完整回归测试再动）
- 每轮关卡布局改成随机生成后，还没有请用户实机确认新的垫脚箱子攀爬节奏、老鹰不同高度的手感是否都合适
